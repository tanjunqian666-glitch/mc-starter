## 第四轮（02:56-03:08）— walk 兼容性深度排查: syso / mainLoop / Go 1.26

### 核心发现：syso 是 walk GUI 能运行的关键

**syso 的作用**：walk 依赖 Windows Common Controls 6.0（manifest 中指定）以正常创建窗口。syso 文件嵌入 `gui.manifest` 内容到 PE 文件的资源中，使得 Windows 加载 `comctl32.dll` v6.0 而非 v5.0。**没有 syso 的独立 walk 程序在 Windows 11 LTSC 上静默闪退（CreateWindowEx 失败，无弹窗）**。这是 `test_walk.go` / `test_walk2.go` / `test_vanilla.go` 全都不显示窗口的根本原因。

**关键证据**：
- 旧 `build/starter-gui.exe`（4月30日编译）能正常弹出窗口 — syso 在 `internal/gui/` 目录内，所有构建正确包含
- `test_walk2.go` / `test_vanilla.go` — 独立 main 包，不在 `internal/gui/` 目录，没有 syso → 不弹窗是预期行为
- walk 库的 `mainLoop()` 实现有两种：纯 Go（默认）和 C `walk_use_cgo` 标签 — 但这两种 `mainLoop` 都需要窗口句柄存在，没有 syso 导致窗口创建失败，`mainLoop` 走不到

### syso 文件位置规则
- Go 的 `-H windowsgui` 嵌入的是 Go 自己的 manifest，不包含 comctl32 v6.0
- syso 必须与引用它的 Go 源码在**同一包目录**才能被编译器自动嵌入到最终 exe
- 项目 syso 路径: `internal/gui/gui_windows_amd64.syso`
- 旧的 `rsrc_windows_amd64.syso` 在项目根目录已作废

### walk 的 mainLoop 实现选择
- `mainloop_cgo.go` — 标签 `// +build windows,walk_use_cgo`，用 C 的 `GetMessage` 消息泵
- `mainloop_default.go` — 标签 `// +build windows,!walk_use_cgo`，纯 Go `GetMessage` + `TranslateMessage` + `DispatchMessage`
- Go 1.26.2 对旧 `// +build` 标签仍支持，但 `walk_use_cgo` 非必须
- **两个版本都能用**，前提是有 syso 嵌入 Common Controls 6.0
