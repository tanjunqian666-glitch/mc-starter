# MC 版本更新器 — 模组与配置同步策略

> 补充方案：模组的增删改怎么管理、配置文件怎么覆盖、更新频率怎么控制

---

## 一、同步体系总览

更新器管理的文件分为两大类，策略不同：

| 类型 | 内容 | 同步策略 |
|---|---|---|
| **模组 (mods/)** | `.jar` 文件 | **增量 + 差异同步** |
| **配置文件 (config/)** | `.toml`, `.json`, `.properties` 等 | **全量覆盖 / 按文件覆盖** |
| **资源包 (resourcepacks/)** | `.zip` | **增量（同模组策略）** |
| **shader 包 (shaderpacks/)** | `.zip` | **增量（同模组策略）** |

---

## 二、模组同步策略

### 2.1 核心原则

```
服务端是唯一真相来源（Source of Truth）。
用户可以在客户端自己加模组，但下次同步时会被服务端清单覆盖。
```

### 2.2 Packwiz 格式下的同步流程

packwiz 使用 `index.toml` 作为模组清单，每个模组有一个 `.pw.toml` 元数据文件。

```
pack.toml             ← 整合包根配置（版本、依赖）
  └── index.toml      ← 模组清单（所有文件的列表）
        ├── mods/sodium.pw.toml      ← 模组 A 元数据
        ├── mods/lithium.pw.toml     ← 模组 B 元数据
        ├── config/ XYZ.toml         ← 配置文件
        └── resourcepacks/...pw.toml ← 资源包
```

**同步算法**：

```go
func SyncMods(modpackDir string, mcDir string) error {
    // 1. 下载最新的 pack.toml + index.toml
    // 2. 解析 index.toml 得到"服务端期望的文件清单"
    remoteFiles := parseIndex(indexPath)
    
    // 3. 扫描本地 mods/ 目录
    localFiles := scanDir(filepath.Join(mcDir, "mods"), "*.jar")
    
    // 4. 差异计算
    toDownload := []string{}  // 服务端有、本地没有
    toDelete   := []string{}  // 本地有、服务端没有（需要删除）
    toUpdate   := []string{}  // 服务端 hash 与本地 hash 不同
    
    for _, rf := range remoteFiles {
        lf := localFiles[rf.Name]
        if lf == nil {
            toDownload = append(toDownload, rf)
        } else if lf.Hash != rf.Hash {
            toUpdate = append(toUpdate, rf)
        }
    }
    
    for name := range localFiles {
        if remoteFiles[name] == nil {
            toDelete = append(toDelete, name)
        }
    }
    
    // 5. 执行操作
    // 删除：从本地删除多余的模组
    for _, name := range toDelete {
        os.Remove(filepath.Join(mcDir, "mods", name))
        log("删除了本地多余的模组: %s", name)
    }
    
    // 下载/更新：下载新模组
    for _, f := range append(toDownload, toUpdate...) {
        download(f.URL, filepath.Join(mcDir, "mods", f.Name), f.Hash)
    }
    
    // 6. 清理残留的 .pw.toml 元数据文件（packwiz 用的，用户不需要）
    // 这些文件放在 modpacks 目录下，不出现在 .minecraft/mods/ 中
}
```

### 2.3 没有 packwiz 的简化方案

如果不打算用 packwiz，可以自建一个简化的模组清单：

**server.json 中定义：**
```json
{
  "mods": [
    {
      "name": "sodium-fabric-0.5.3+mc1.20.1.jar",
      "url": "https://cdn.modrinth.com/data/.../sodium-0.5.3.jar",
      "hash": "sha256:abc...",
      "required": true
    },
    {
      "name": "lithium-fabric-0.11.2+mc1.20.1.jar",
      "url": "https://cdn.modrinth.com/data/.../lithium-0.11.2.jar",
      "hash": "sha256:def...",
      "required": true
    }
  ]
}
```

**同步算法相同**：对比 server.json 清单 vs 本地 mods/ 目录 → 差异化处理。

## 三、配置同步策略

配置文件比模组更敏感——用户可能手动改过配置，全量覆盖会丢设置。

### 3.1 配置类型分类

| 类型 | 例子 | 行为 |
|---|---|---|
| **服务端强制配置** | `server.properties`, `ops.json` | 每次强制覆盖 |
| **推荐配置** | 模组配置文件 `.toml/.json` | 首次安装时写入，后续可选更新 |
| **用户配置** | `options.txt`, `screenshots/` | 不碰 |

### 3.2 三种覆盖模式

```json
// server.json
{
  "configs": [
    {
      "path": "config/sodium-mixins.properties",
      "url": "https://...",
      "hash": "sha256:...",
      "mode": "force"           // force / once / prompt
    },
    {
      "path": "config/jei-server-settings.cfg",
      "url": "https://...",
      "mode": "once"            // 仅首次安装时写入
    },
    {
      "path": "config/iris.properties",
      "url": "https://...",
      "mode": "prompt"          // 每次询问用户
    }
  ]
}
```

**三种模式详解**：

| 模式 | 行为 | 适用场景 |
|---|---|---|
| `force` | 每次同步都覆盖，不管本地有没有改过 | 服务器规则、反作弊配置 |
| `once` | 仅当文件不存在时写入，已有就不动 | 模组默认配置（用户可能调过） |
| `prompt` | 服务端有更新时询问用户是否覆盖 | 重要的配置文件变更 |

### 3.3 配置文件版本追踪

为了实现 `once` 和 `prompt`，需要记录配置文件状态：

```go
// updater/cache/config_state.json
{
  "version": "1.0.0",
  "configs": {
    "config/sodium-mixins.properties": {
      "last_hash": "sha256:oldhash",
      "first_installed": "2026-04-29T02:00:00Z",
      "mode": "once"
    },
    "config/jei-server-settings.cfg": {
      "last_hash": "",
      "first_installed": "2026-04-29T02:00:00Z",
      "mode": "once"
    }
  }
}
```

```go
// 配置文件同步逻辑
func syncConfig(remote ConfigEntry, localState map[string]ConfigState, mcDir string) error {
    localPath := filepath.Join(mcDir, remote.Path)
    state := localState[remote.Path]
    
    switch remote.Mode {
    case "force":
        // 不管三七二十一，覆盖
        download(remote.URL, localPath, remote.Hash)
        
    case "once":
        // 仅首次
        if _, err := os.Stat(localPath); os.IsNotExist(err) {
            download(remote.URL, localPath, remote.Hash)
        }
        // 已存在就不动
        
    case "prompt":
        // 检查 hash 是否变化
        currentHash := computeHash(localPath)
        if currentHash != remote.Hash && state.LastHash != remote.Hash {
            // 有更新，询问用户
            if promptUser("配置文件 %s 有更新，是否覆盖？", remote.Path) {
                download(remote.URL, localPath, remote.Hash)
            }
        }
    }
    
    // 更新本地状态记录
    state.LastHash = remote.Hash
    return nil
}
```

### 3.4 UPMC 的做法（参考）

UPMC 的处理方式：

```
updater/settings-download.zip  ← settings 打包成 zip
  └── 解压到 .minecraft/config/

updater/.settings_installed    ← 标记文件（表示已安装过）
```

- **首次**：下载 zip → 解压覆盖 → 写标记文件
- **后续**：检测标记文件存在 → 跳过
- **手动删除标记文件** → 下次重新解压

这是个"一刀切"的 `once` 模式，配置包整体只安装一次。

## 四、版本间差异处理

### 4.1 换版本时模组冲突

当 `server.json` 的 `mc_version` 从 `1.20.1` 改成 `1.21` 时：

```go
// 检测版本变更
func detectVersionChange(oldCfg, newCfg *ServerConfig) []Action {
    if oldCfg.MCVersion != newCfg.MCVersion {
        // MC 版本变了 → 模组大概率不兼容
        return []Action{
            {Type: "warn", Msg: "MC 版本从 1.20.1 变更为 1.21，所有 Fabric 模组将重新下载"},
            {Type: "clear_mods", Target: "mods/"},
            {Type: "backup_config", Target: "config/"},
            {Type: "sync", Target: "all"},
        }
    }
    
    if oldCfg.Loader.Version != newCfg.Loader.Version {
        // 仅 Loader 版本更新
        return []Action{
            {Type: "update_loader", Version: newCfg.Loader.Version},
        }
    }
    
    // 仅模组清单变化 → 差异化同步
    return []Action{
        {Type: "diff_sync", Target: "mods/"},
    }
}
```

**换版本时建议行为**：
1. 备份旧 mods/ 到 `mods_backup_{version}/`
2. 清空 mods/（防止不兼容模组导致 MC 崩溃）
3. 重新下载新版本的模组
4. 配置文件：保留用户配置，只覆盖兼容的部分

### 4.2 模组增删

**新增模组** → 直接下载，下次同步时用户看到新模组生效

**删除模组** → 从服务端清单移除，下次同步时：
- 本地对应文件被删除
- 用户加的额外模组不受影响（只删清单里的，不碰清单外的）

**但客户端自定义模组问题：**
```
用户自己加了光影包模组（Oculus），不在 server.json 清单里
→ 同步后：Oculus 还在（因为没出现在服务端清单里，不会触发删除）
→ 但如果服务端切换了 MC 版本，清空 mods/ → Oculus 也会被删
```

解决方案：版本大变更时弹提示：
```
[!] MC 版本从 1.20.1 变更为 1.21
[!] 将清空 mods/ 目录并重新下载
[!] 用户自定义模组（Oculus 等）将被移除
[?] 已备份到 mods_backup_1.20.1/，请手动确认兼容性
```

## 六、配置建议（对你当前项目）

### MVP 阶段

```
模组：差异化同步（对比 hash，增删改）
配置：force 模式 + 受控的配置包
版本变更：清空 mods/ + 重新下载 + 备份旧文件
```

具体实现就先简化成：

```json
// server.json
{
  "mods": [
    // 模组清单，同步时差异计算
  ],
  "config_zip": "https://.../config.zip",   // 配置包
  "config_mode": "once",                      // 仅首次安装
  "config_hash": "sha256:..."
}
```

同步时：
1. 下载 `config_zip` → 校验 hash → 如果配置模式是 `once` 且本地已有 → 跳过
2. 对比模组清单 → 增删改

这样 MVP 阶段配置同步只需要一个 zip 包 + 一个标记文件，简单稳定。

后续再扩展 `force` / `prompt` 等精细控制。
