# MC 版本更新器 — 服务端整合包管理流程

> **纯服务端功能**：服务器解包、解析、差异分析、版本发布
> 管理员只需上传整合包 zip，工具自动完成导入 → 差异预览 → 发布

---

## 一、整体流程

```
┌──────────────────────────────────────────────────────────────┐
│                        管理员                                  │
│  拿到整合包 zip → 放到服务器 → 运行导入命令                   │
└─────────────────────────┬────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────────┐
│                        starter import                           │
│                                                              │
│  ① 解包：解析 zip 内容                                       │
│     └→ 提取文件清单（路径、hash、大小）                       │
│                                                              │
│  ② 分析：扫描模组、配置、资源包等                             │
│     ├─ mods/*.jar  → 列表 + hash                           │
│     ├─ config/*    → 列表 + hash                           │
│     └─ 识别 MC 版本和 Loader（从模组推断）                   │
│                                                              │
│  ③ 对比：与当前发布版本的差异                                 │
│     └→ 新增了什么、删了什么、更新了什么                      │
│                                                              │
│  ④ 输出：差异预览 + 待发布版本（draft）                      │
│     └→ 管理员审查确认后再 publish                             │
│                                                              │
└─────────────────────────┬────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────────┐
│                      starter publish                           │
│                                                              │
│  ① 将 draft 转 published                                    │
│  ② 生成增量清单（对比上一版本）                               │
│  ③ 更新 server.json                                           │
│  ④ 更新 current symlink                                       │
│                                                              │
└─────────────────────────┬────────────────────────────────────┘
                          │
                          ▼
┌──────────────────────────────────────────────────────────────┐
│                        客户端                                  │
│  请求 /latest?from=1.1.0 → 拿到增量清单 → 按 hash 下载文件    │
│  只拉变更过的文件，不整包下载                                  │
└──────────────────────────────────────────────────────────────┘
```

## 二、CLI 命令

### 2.1 import — 导入整合包 zip

```bash
# 导入一个整合包 zip，与当前版本对比
starter import ./cjc-pack-v1.2.0.zip

# 指定发布目录
starter import ./cjc-pack-v1.2.0.zip --repo /data/mc-starter/repo
```

**输出预览**：

```
=== 导入整合包 ===
zip: cjc-pack-v1.2.0.zip (50.2 MB, 38 个文件)
MC 版本: 1.20.1 (从模组推断)
Loader: fabric 0.15.11

差异对比 (相对于 v1.1.0):
  + iris-1.6.4+mc1.20.1.jar                4.0 MB  新增
  - optifine-1.20.1.jar                    2.0 MB  删除
  ~ sodium-fabric-0.5.3+mc1.20.1.jar      5.0 MB  更新
  ~ config/sodium-mixins.properties        0.5 KB  更新

增量大小: 9.0 MB (全量 50 MB 的 18%)
状态: [draft] 待发布

运行 starter publish 确认发布
```

### 2.2 publish — 发布版本

```bash
# 发布当前 draft
starter publish

# 指定版本号和发布消息
starter publish --version 1.2.0 --message "更新模组版本"
```

### 2.3 diff — 查看两个版本的差异

```bash
# 比较两个已发布的版本
starter diff v1.0.0 v1.1.0

# 对比已导入的 draft 和当前发布版本
starter diff draft current
```

### 2.4 rollback — 回滚版本

```bash
# 回滚到上一版本
starter rollback

# 回滚到指定版本
starter rollback v1.0.0
```

### 2.5 list — 查看版本历史

```bash
starter list
```

## 三、差异分析逻辑

### 3.1 文件清单对比

```go
type FileEntry struct {
    Path    string // 相对路径，如 "mods/sodium.jar"
    Hash    string // SHA256 hex
    Size    int64  // 字节数
}

type VersionDiff struct {
    Added      []FileEntry // 新版本有、旧版本无
    Removed    []FileEntry // 旧版本有、新版本无
    Updated    []FileEntry // 都有但 hash 不同（含旧 hash 字段）
    Unchanged  int         // 未变化的文件数
}

func computeDiff(prev, curr []FileEntry) *VersionDiff {
    // 构建索引
    prevIdx := make(map[string]FileEntry)
    for _, e := range prev { prevIdx[e.Path] = e }
    currIdx := make(map[string]FileEntry)
    for _, e := range curr { currIdx[e.Path] = e }

    diff := &VersionDiff{}
    for path, entry := range currIdx {
        if old, ok := prevIdx[path]; !ok {
            diff.Added = append(diff.Added, entry)
        } else if old.Hash != entry.Hash {
            diff.Updated = append(diff.Updated, entry)
        }
    }
    for path := range prevIdx {
        if _, ok := currIdx[path]; !ok {
            diff.Removed = append(diff.Removed, prevIdx[path])
        }
    }
    return diff
}
```

### 3.2 差异结果显示

```go
func (d *VersionDiff) Summary() string {
    return fmt.Sprintf("+%d 新增, -%d 删除, ~%d 更新, =%d 未变",
        len(d.Added), len(d.Removed), len(d.Updated), d.Unchanged)
}
```

## 四、发布仓库结构

```
/repo/                         ← 服务端发布目录
├── packs/
│   └── cjc-pack/
│       ├── meta.json          ← 包元信息（名称、当前版本等）
│       ├── versions/
│       │   ├── v1.0.0/
│       │   │   ├── manifest.json   ← 文件清单
│       │   │   └── files/          ← 按 hash 去重的文件存根
│       │   ├── v1.1.0/
│       │   │   ├── manifest.json
│       │   │   ├── diff_from_v1.0.0.json
│       │   │   └── files/
│       │   └── v1.2.0.draft/      ← 待发布版本
│       │       ├── manifest.json
│       │       └── diff_from_v1.1.0.json
│       └── current → versions/v1.1.0
│
├── server.json                ← 客户端拉取的入口文件
│                                (包含版本号、MC 版本、Loader、下载方式)
│
└── cdn/                       ← CDN 文件分发目录（可选）
    └── files/                 ← 按 hash 命名的文件
```

## 五、与客户端的关系

```
服务器                                             客户端
──────                                            ──────

starter import pack.zip      │
  └→ 解包分析 + diff        │
  └→ 生成 draft             │
                             │
starter publish              │
  └→ draft → published      │
  └→ 更新 server.json       │
  └→ 更新 current symlink   │
                             │
                             │      GET /server.json
                             │    ← {"version": "1.2.0", ...}
                             │
                             │      GET /api/latest?from=1.1.0
                             │    ← {"added": [...], "removed": [...], ...}
                             │
                             │      按 hash 下载变更文件
                             │    ← 增量更新完成
```

## 六、旧版本兼容

传统全量 zip 方案也保留：

```json
// server.json
{
  "version": "1.2.0",
  "mc_version": "1.20.1",
  "loader": { "type": "fabric", "version": "0.15.11" },
  "pack_url": "https://cdn.example.com/pack-v1.2.0.zip",   // 全量
  "pack_hash": "sha256:..."
}
```

增量方案上线后，`server.json` 变为：

```json
{
  "version": "1.2.0",
  "from_version": "1.1.0",
  "mc_version": "1.20.1",
  "loader": { "type": "fabric", "version": "0.15.11" },
  "update_mode": "incremental",
  "incremental": {
    "url": "https://cdn.example.com/api/update?from=1.1.0",
    "added": [...],
    "removed": [...],
    "updated": [...]
  },
  "full_pack": {   // 兜底
    "url": "https://cdn.example.com/pack-v1.2.0.zip",
    "hash": "sha256:..."
  }
}
```

两种模式共存，客户端根据本地情况自动选择。
