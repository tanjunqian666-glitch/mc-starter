# 服务端整合包管理流程（更新：HTTP API + 多包模式）

> 本文档关联：`internal/pack/pack.go`、服务端 REST API

---

## 一、整体流程（管理员视角）

```
上传 zip 包:
  POST /api/v1/admin/packs/{name}/import
  Content-Type: multipart/form-data
  → 服务端解包扫描
  → 与上一版本对比差异
  → 返回差异预览

审查差异:
  → 管理员确认无误

发布:
  POST /api/v1/admin/packs/{name}/publish
  Body: {"message":"更新模组","primary":true}
  → draft → published
  → 更新增量清单

客户端同步:
  GET /api/v1/packs/{name}/update?from=v1.1.0
  → 返回 added/updated/removed
  → 客户端按 hash 下载文件
```

## 二、CLI（管理员本地操作）

旧版 CLI 操作仍然保留，用于本地开发测试：

```bash
# 本地导入
starter pack import ./cjc-pack-v1.2.0.zip

# 本地发布
starter pack publish --version v1.2.0 --message "更新模组"
```

## 三、已发布的仓库结构

```
packs/
└── {name}/
    ├── meta.json            ← 包元信息（名称/RSA primary 标记等）
    ├── server.json          ← 供客户端 GET /packs/{name} 使用
    ├── versions/
    │   ├── v1.0.0/
    │   │   ├── manifest.json   ← 文件清单（兼容旧格式）
    │   │   └── diff.json       ← 增量清单（有上一版本时）
    │   ├── v1.1.0/
    │   │   ├── manifest.json
    │   │   └── diff.json
    │   └── v1.2.0.draft/       ← 待发布
    ├── files/                   ← 按 hash 存储的文件
    └── current → versions/v1.1.0  ← symlink
```

## 四、增量更新机制

与原有逻辑一致（`pack.ComputeDiff`），只是调用方从 CLI 变为 HTTP handler：

```
管理员 publish → 服务端:
  1. 读 draft manifest
  2. 找到上一版 manifest
  3. ComputeDiff(prev, curr) → added/updated/removed
  4. 文件移入 files/{hash_prefix}/{hash}
  5. 写入 diff.json + 更新 server.json
  6. 更新 packs/index.json

客户端 update → HTTP:
  1. GET /packs/{name}/update?from=v1.1.0
  2. 服务端读取 v1.1.0 manifest + latest manifest
  3. ComputeDiff → 返回 JSON
  4. 客户端按 hash 下载文件
```

## 五、旧版本兼容

- 支持只发全量包（mode: "full"），不生成增量
- 全量包放在 files/ 下，客户端按需整包下载
- 增量/全量在 server.json 中标记
