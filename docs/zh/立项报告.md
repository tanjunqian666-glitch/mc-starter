# MC 版本更新器 — 项目立项报告

> 基于 UPMC 逆向经验，取其精华，去其糟粕

## 一、项目背景

### 1.1 目标

开发一个轻量级的 **Minecraft 版本管理 + 整合包更新器**，聚焦 MC 版本管理这一核心职能，不做代理、不杀进程、不捆绑额外功能。

### 1.2 为什么自己不做完整启动器

MC 生态已有成熟的启动器：
- **PCL2**（Plain Craft Launcher 2）— 功能最全的社区启动器
- **HMCL**（Hello Minecraft! Launcher）— 开源跨平台
- **官方启动器** — 省心但慢

我们不需要重复造轮子。我们需要的是 **整合包/服务器专属的版本管理工具**：自动下载指定版本的 MC + Fabric/Forge + 模组包，然后交给用户选择的启动器去运行。

### 1.3 与 UPMC 的区别

| | UPMC（逆向对象） | 本项目 |
|---|---|---|
| 代理 | **强制捆绑 Xray** | 不做 |
| 杀进程 | 强杀 Discord | 不做 |
| MC 启动 | 捆绑 PCL2 | 不负责（交由第三方启动器如 PCL2/HMCL/Prism） |
| 自更新 | 有 | 有（借鉴） |
| 整合包 | packwiz | packwiz（借鉴） |
| 模组管理 | 自动全流程 | 自动全流程（借鉴） |
| Java 管理 | 自带下载 JRE 21 | 可选（检测/提示/下载） |
| 技术栈 | Rust + rustls | Go（推荐） |

## 二、技术选型

### 2.1 推荐技术栈

| 层 | 选型 | 理由 |
|---|---|---|
| 语言 | **Go** | 交叉编译简单、TLS 库友好（crypto/tls 可插桩自定义 CA）、编译单文件 |
| GUI | **无**（CLI + 可选 Web UI） | 核心功能不需要 GUI，CLI 更稳定。如果要做 UI，用 PCL2 或 HMCL 前端 |
| 模组包格式 | **packwiz**（pack.toml） | 行业标准，增量更新，成熟 |
| 存储 | **本地 JSON**（借鉴 upmc 的 channel.json/local.json） | 轻量，可读 |
| HTTP | **Go 标准库 net/http** | 够用 |
| 更新 | **自更新模式**（借鉴 Chrome/upmc 的 .exe.new 替换模式） | 成熟稳定 |

### 2.2 为什么 Go 优于 Rust

1. **TLS 控制**：Go 的 crypto/tls 可以注入自定义根证书、跳过验证、做 MITM 代理，而 Rust 的 rustls 固化 webpki 根证书库，很难插桩
2. **编译速度**：Go 秒级编译，Rust 分钟级
3. **交叉编译**：Go `GOOS=windows GOARCH=amd64 go build` 一行搞定，Rust 需要配工具链
4. **依赖管理**：Go 标准库"batteries included"，HTTP/TLS/压缩/JSON 全内置

如果未来要做 Web UI，Go 可以自嵌一个单页应用；Rust 则需要额外 crate。

## 三、功能范围

### MVP（v1.0）

```
检测本地 Java → 读取 server.json 配置 → 
获取远程版本清单 → 下载 MC jar + asset + library →
下载 Fabric/Forge libraries → 通过 packwiz 同步模组 →
输出标准 .minecraft 目录（PCL2/HMCL 直接识别）
```

#### 具体功能

**1. MC 版本管理**
- 从 Mojang 版本清单 API 拉取版本列表
- 增量下载（只下载缺失的 asset/version jar）
- 支持多版本共存（`.minecraft/versions/`）

**2. Fabric/Forge 依赖下载**
- 从 BMCLAPI 镜像获取 Fabric meta profile
- 解析 libraries 列表并下载到 `.minecraft/libraries/`
- 仅负责资源同步，不生成 JVM 参数、不执行 java

**3. 整合包同步（packwiz / zip）**
- pack.toml 解析 / zip 解包扫描
- 增量模组下载/更新
- 配置文件覆盖

**4. Java 环境检测**
- 查找系统 Java（JAVA_HOME/PATH/注册表）
- 可选：下载 Adoptium JRE（清华镜像）

**5. 启动器兼容**
- 输出标准 `.minecraft` 目录（MultiMC 规范对齐）
- PCL.ini 注入（Version/VersionCache）
- PCL2/HMCL/Prism/官方启动器可直接识别

### v2.0（可选项）

**1. GUI 前端**
- 轻量 Web UI 或嵌入 PCL2
- 进度条、日志展示

**2. 多通道支持**
- stable / beta / dev 三通道（借鉴 channel.json）
- 通道可配置，服务端控制

**3. 自定义镜像源**
- 用户可指定 Maven/Fabric/MC 的镜像地址
- 内置国内镜像 fallback 列表

**4. 自更新**
- 启动时检查版本
- 增量下载更新

### 不做（非功能）

- ❌ 代理/VPN 功能
- ❌ 杀进程
- ❌ 账户登录
- ❌ 皮肤管理
- ❌ 社交功能

### 4.2 配置结构

借鉴 upmc 的 server.json + local.json 双文件设计：

**server.json（服务端下发，只读）**
```json
{
  "version_tag": "stable",
  "mc_version": "1.20.1",
  "loader": {
    "type": "fabric",
    "version": "0.15.11"
  },
  "modpack": {
    "url": "https://example.com/pack.toml",
    "hash": "sha256:..."
  },
  "java": {
    "min_version": 17,
    "download_url": "https://mirrors.tuna.tsinghua.edu.cn/Adoptium/21/jre/x64/windows/"
  },
  "mirrors": {
    "fabric_meta": "https://bmclapi2.bangbang93.com/fabric-meta/",
    "fabric_maven": "https://bmclapi2.bangbang93.com/maven/"
  }
}
```

**local.json（本地读写）**
```json
{
  "install_path": "./.minecraft",
  "launcher": "pcl2",
  "java_home": "",
  "memory": "4096",
  "username": "",
  "mirror_mode": "auto"
}
```

### 4.3 启动流程

```
[Check] Java 环境存在？
  ├─ No  → 提示安装 / 可选自动下载
  └─ Yes → 检查版本 ≥ min_version
            ├─ No  → 提示升级
            └─ Yes → 继续

[Sync] 获取 server.json
  ├─ 失败 → 使用本地缓存版本
  └─ 成功 → 缓存到本地

[Version] 检查 MC jar 是否存在
  ├─ No  → 从 Mojang API / 镜像下载
  └─ Yes → 跳过

[Loader] 检查 Fabric/Forge 是否已安装
  ├─ No  → 下载安装器 + 安装
  └─ Yes → 跳过

[Modpack] packwiz 同步
  └─ 增量更新模组

[Launch / Done] 用户打开 PCL2 → 点启动
```

## 五、借鉴 UPMC 的好设计

### 要做

1. **配置驱动** — server.json 控制一切，服务端改配置=改客户端行为，无需发版 ✅
2. **packwiz 模组包** — 行业标准，增量更新，好用 ✅
3. **国内镜像** — BMCLAPI + 清华镜像，国内用户友好 ✅
4. **自更新** — `.exe.new` 替换模式，稳定可靠 ✅
5. **PCL2 兼容** — 与主流启动器深度整合，用户无感 ✅
6. **降级容忍** — 服务器不可用时用缓存，不影响正常使用 ✅

### 不做

1. **裸启动 / 生成 JVM 参数** — 由 PCL2/HMCL 完成 ❌
2. **捆绑代理** — 用户可选独立代理工具 ❌
3. **强杀进程** — 弹窗询问，而不是静默 kill ❌
4. **单 CDN 依赖** — 内置多 CDN 回退列表 ❌
5. **Rust + rustls** — Go 的 TLS 更好控制 ❌
6. **GUI 过重** — 保持 CLI 优先，GUI 可选 ❌

| P2 | Fabric libraries + 修复栈 + daemon | 2-3 天 |
| P3 | Java 检测 | 1 天 |

## 七、风险

| 风险 | 缓解 |
|---|---|
| Mojang API 变更 | 使用主流镜像缓存层 |
| packwiz 格式演进 | 锁定 packwiz 版本，定期跟进 |
| BMCLAPI 不可用 | 内置多镜像回退 |
| 用户无 Java | 提供自动下载选项（清华镜像） |
| Rust 锈住不会了 | 选 Go，资源少但资料多 |
