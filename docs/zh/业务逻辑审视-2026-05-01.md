# 业务逻辑审视报告 — 2026-05-01

> 基于 PCL2 整合包安装流程对比，以"用户新电脑一键安装即玩"为价值基准，审视 mc-starter 当前业务逻辑。

---

## 一、核心缺陷：三条独立链路未打通

### 用户期望（正确的端到端流程）

```
新电脑 → 配 API → 选目录 → 选版本 → 点更新
  ↓
1. 下载 MC 本体（client.jar + libraries + assets）          ← sync
2. 安装 Loader（fabric/forge/neoforge，指定 mc 版本）      ← fabric install
3. 下载整合包自定义内容（mods/config/resourcepacks）        ← update
4. 组合成完整可用版本
5. 启动游戏
  ↓
✅ 能玩
```

### 实际现状（碎片化的三条链路）

| 链路 | 命令 | 负责什么 | 当前状态 |
|------|------|----------|----------|
| A | `starter sync` | MC 本体（vanilla） | ✅ 能下，但只下原版，不下 loader |
| B | `starter fabric install` | Loader 安装 | ✅ 能装，但用户必须手动指定 mc 版本+loader 版本 |
| C | `starter update` | 整合包自定义内容 | ✅ 能同步，但只关心 packs/，不关心 MC 本体和 loader |

**链路 A/B/C 互不知晓对方的存在。** 没有任何一条命令能跑完 A→B→C 全流程。

### 具体问题清单

#### P0 — 用户无法一键安装

1. **`starter run` 只跑了 sync + update，没装 loader**
   - 第 273 行 `// TODO: 5. 拉起启动器` 是空的
   - 没调用 `fabric install`
   - 没从服务端获取 `(mc_version, loader)` 信息

2. **`starter run` 的目标版本确定逻辑混乱**
   - 优先读 `server.json`（本地文件，通常不存在）
   - 回退到拉 Mojang manifest 取最新 release
   - **但整合包用户并不在意 MC 原版版本号，他们在意的是包名+包版本**

3. **`starter update` 和整合包版本的关联缺失**
   - update 从服务端拉增量清单，但返回的 `IncrementalUpdate` 没有 `mc_version` 和 `loader` 字段
   - 客户端 update 完成后不知道这个包需要什么 MC 版本、什么 Loader

4. **`starter fabric install` 需要用户手动指定参数**
   - 用户必须知道包需要 `fabric 0.16.10` + `MC 1.21.1`
   - 修复模式 `--loader-only` 最后也只是打印"请运行 fabric install"，没自动完成

#### P1 — 设计问题

5. **packs/ 和 versions/ 之间无联动**
   - packs/ 下放的是 mods/config
   - versions/ 下是 MC 本体版�的 jar + json
   - 用户最后要手动把 packs 的内容复制/合并到 versions 才能玩

6. **`finder.go` 查找 MC 目录的逻辑与 C/S 模式冲突**
   - `FindManagedVersions` 试图在本地扫描 PCL2 配置/目录来定位版本
   - 但 C/S 模式下，目录应该是配置里写死的，不需要扫描猜测

## 二、修复方案

// IncrementalUpdate（客户端增量 API 响应）

    // ... 现有字段

## 三、过时/无效的代码和能力

### 3.1 `finder.go` 中的目录扫描逻辑

`FindManagedVersions` 通过扫描 PCL2 配置 + 默认路径来猜测 MC 目录。在 C/S 模式下，目录应来自配置，不应猜测。

**问题**：`run()` 和 `sync()` 里大量使用 `finder.FindManagedVersions` 做目录判断，这在新电脑场景下只会返回"未找到"，然后用户还得手动配。

**建议**：降级为纯配置驱动，finder 只做校验（目录是否存在、version.json 是否有效），不做"猜目录"。

### 3.2 `ensureConfig` — 默认配置空转

`ensureConfig` 创建默认配置但用不了——`MinecraftDir` 默认 `.minecraft`、`Launcher` 默认 `bare`、`Username` 默认 `Player`。用户必须手动编辑。

**建议**：改为 GUI 交互引导，或至少打印清晰的下一步提示。

### 3.3 部分测试过时

现有测试（特别是 `finder_test.go`、`incr_sync_test.go`）测试的是"猜目录"逻辑和旧增量同步流程。C/S 模式改了后这些测试可能还在跑旧的假设。
