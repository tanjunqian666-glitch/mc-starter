# MC 版本更新器 — 整合包打包与导入方案（服务端）

> 服务端导入 zip 整合包，对比现有版本差异，生成待发布版本
> **这是服务端工作流，客户端不直接解压 zip**

---

## 一、服务端发布流程

### 1.1 推荐格式：普通 zip

```
cjc-pack-1.2.0.zip
├── mods/                  ← 模组
│   ├── sodium-0.5.3.jar
│   └── lithium-0.11.2.jar
├── config/                ← 配置文件
│   └── sodium-mixins.properties
├── resourcepacks/         ← 资源包（可选）
├── shaderpacks/           ← 光影包（可选）
├── kubejs/                ← KubeJS 脚本（可选）
├── defaultconfigs/        ← 全局配置（可选）
└── pack.png               ← 整合包图标（可选）
```

**特别说明**：

- **不需要** `versions/` 目录（MC 版本和 Loader 由配置单独管理）
- **不需要** `PCL/Setup.ini`
- **不包含** MC 本体、assets、libraries（这些由客户端从镜像下载）
- 模组 jar 文件**全部内嵌**（不依赖 Modrinth/CurseForge API）
- 服务端目录支持 `serverconfig/` 等额外内容

### 1.2 服务端导入步骤

```
1. 从 CurseForge / Modrinth / PCL2 导出得到整合包 zip

2. 运行导入命令：
   starter import ./cjc-pack-v1.2.0.zip

   程序自动完成：
   ├─ 解包分析（模组清单、配置清单、MC 版本、Loader）
   ├─ 与当前发布版本做差异对比
   ├─ 生成差异预览（新增/删除/更新了哪些文件）
   └─ 保存为"待发布版本"（draft 状态）

3. 审查差异预览

4. 确认发布：
   starter publish                # 将 draft 转为 published
   starter publish --version 1.2.0 --message "更新模组版本"
```

### 1.3 版本命名

```
cjc-pack-1.0.0.zip    ← 初始版本
cjc-pack-1.1.0.zip    ← 加模组/改配置
cjc-pack-1.2.0.zip    ← 更新模组
```

服务端保留所有历史版本的 manifest（供增量对比使用）。

## 二、服务端导入与差异分析

### 2.1 导入算法

```go
func importPack(zipPath string, repoDir string) (*ImportResult, error) {
    // 1. 解压 zip 到临时目录
    tmpDir := extractToTemp(zipPath)
    defer os.RemoveAll(tmpDir)

    // 2. 扫描文件清单（路径 + SHA1 + 大小）
    newManifest := scanFiles(tmpDir, "")

    // 3. 加载上一版本的 manifest（如有）
    prevManifest := loadLatestManifest(repoDir)

    // 4. 计算差异
    diff := computeDiff(prevManifest, newManifest)

    // 5. 返回预览结果（不写入工作目录）
    return &ImportResult{
        Diff:        diff,
        NewManifest: newManifest,
        Stats:       diffStats(diff),
        Status:      "draft",
    }, nil
}
```

### 2.3 差异分析关键代码

```go
func diffVersions(prev, curr *PackManifest) *VersionDiff {
    diff := &VersionDiff{}

    prevFiles := make(map[string]FileEntry)
    for _, f := range prev.Files {
        prevFiles[f.Path] = f
    }
    currFiles := make(map[string]FileEntry)
    for _, f := range curr.Files {
        currFiles[f.Path] = f
    }

    for path, entry := range currFiles {
        if old, ok := prevFiles[path]; !ok {
            diff.Added = append(diff.Added, entry)
        } else if old.Hash != entry.Hash {
            diff.Updated = append(diff.Updated, entry)
        }
    }
    for path, entry := range prevFiles {
        if _, ok := currFiles[path]; !ok {
            diff.Removed = append(diff.Removed, entry)
        }
    }

    return diff
}
```

## 三、发布流程

### 3.1 发布动作

```go
func publish(repoDir string, draftName string) error {
    // 1. 读取 draft 版本
    draft := loadDraft(repoDir, draftName)

    // 2. 将文件从临时区复制到发布目录
    copyFiles(draft.TempDir, repoDir+"/files/")

    // 3. 生成发布清单（含增量 diff vs 上一版本）
    prevVersion := findLatestPublished(repoDir)
    diff := diffVersions(prevVersion.Manifest, draft.Manifest)
    saveDiff(repoDir, draftName, diff)

    // 4. 更新 server.json（客户端拉取的入口）
    updateServerJSON(repoDir, draft)

    // 5. 将 draft 标记为 published
    markPublished(repoDir, draftName)

    return nil
}
```

### 3.2 发布产出目录

```
/repo/
├── packs/
│   └── cjc-pack/
│       ├── meta.json                  ← 包元信息
│       ├── versions/
│       │   ├── v1.0.0/
│       │   │   ├── manifest.json      ← 该版本文件清单
│       │   │   └── files/             ← 版本文件（按 hash 去重）
│       │   ├── v1.1.0/
│       │   │   ├── manifest.json
│       │   │   ├── diff_from_v1.0.0.json ← 增量清单
│       │   │   └── files/
│       │   └── v1.2.0/               ← 新导入的 draft
│       │       ├── manifest.json
│       │       ├── diff_from_v1.1.0.json
│       │       └── files/
│       └── current → versions/v1.1.0   ← 当前发布版本
│
└── server.json                        ← 客户端拉取的入口配置
```

## 四、客户端更新逻辑（配合服务端）

### 4.1 客户端请求

```
GET /api/mc-starter/v1/latest?from=1.1.0

响应：
{
  "version": "1.2.0",
  "mc_version": "1.20.1",
  "loader": { "type": "fabric", "version": "0.15.11" },
  "update_mode": "incremental",        // full | incremental | none
  "incremental": {
    "added": [
      {"path": "mods/iris-1.6.4.jar",
       "url": "https://cdn.example.com/files/sha256:xxx",
       "hash": "sha256:xxx", "size": 4194304}
    ],
    "removed": ["mods/optifine-1.20.1.jar"],
    "updated": [...]
  },
  "full_pack": {
    "url": "https://cdn.example.com/pack-v1.2.0.zip",
    "hash": "sha256:zzz", "size": 52428800
  }
}
```

### 4.2 更新策略决策

```go
func decideMode(prev, curr *PackMeta, diff *VersionDiff) string {
    if prev.MCVersion != curr.MCVersion { return "full" }
    if prev.Loader != curr.Loader         { return "full" }
    diffSize := diff.AddedSize + diff.UpdatedSize
    if diffSize > curr.TotalSize * 0.5    { return "full" }
    if diffSize == 0                       { return "none" }
    return "incremental"
}
```

- **首次安装** → 全量下载 zip
- **后续小更新** → 增量（只下载变动的文件）
- **MC/Loader 版本变更** → 全量
- **差异超过半包** → 全量（增量性价比低）

## 五、WBS 补充（服务端功能）

| ID | 任务 | 预估 | 前置 |
|---|---|---|---|
| P1.12 | zip 解包 + 文件扫描 + hash 计算 | 3h | P0.x |
| P1.13 | 新旧版本差异分析（added/updated/removed） | 3h | P1.12 |

**注意**：P1.12-P1.14 **纯服务端功能**，不涉及客户端 zip 解压。客户端只通过 API 拉版本信息 + 按 hash 下载文件。

## 六、和传统 PCL2 导出的关系

| 方式 | 产出物 | 优点 | 缺点 |
|---|---|---|---|
| PCL2 导出 .mrpack | `modrinth.index.json` + overrides/ | 模组只保留 URL，包体小 | 需要联网下载模组 |
| 直接 zip | `mods/`, `config/`, ... | 导入即用 | 包体大 |

**服务端方案**：接收 zip → 分析 → diff → 发布增量。管理员只需要上传整合包 zip，后续客户端增量更新自动生成。
