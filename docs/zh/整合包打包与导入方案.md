# MC 版本更新器 — 整合包打包与导入方案

> 服务端发布 zip 整合包，更新器自动同步到版本隔离目录

---

## 一、服务端发布流程

### 1.1 推荐格式：普通 zip

```
cjc-pack-1.2.0.zip
├── mods/                  ← 模组
│   ├── sodium-0.5.3.jar
│   └── lithium-0.11.2.jar
├── config/                ← 配置文件
│   └── sodium-mixins.properties
├── resourcepacks/         ← 资源包（可选）
├── shaderpacks/           ← 光影包（可选）
├── kubejs/                ← KubeJS 脚本（可选）
├── defaultconfigs/        ← 全局配置（可选）
└── pack.png               ← 整合包图标（可选）
```

**特别说明**：

- **不需要** `versions/` 目录（MC 版本和 Loader 由更新器单独处理）
- **不需要** `PCL/Setup.ini`（更新器会生成）
- **不包含** MC 本体、assets、libraries（这些由更新器从镜像下载）
- 模组 jar 文件**全部内嵌**（不需要联网查 Modrinth API 拿下载地址）
- 如果有服务端专用配置目录（如 `serverconfig/`），也可以包含，更新器会按规则处理

### 1.2 服务端发布步骤

```
1. 在 PCL2 / HMCL / 自己整理的文件夹里配好模组和配置

2. 打包为 zip：
    整合包名-版本号.zip
      ├── mods/...
      ├── config/...
      └── ...

3. 计算 hash：
    sha256sum 整合包名-版本号.zip

4. 写 server.json（或者一个简单的 pack.json）：
    {
      "version": "1.2.0",
      "pack_url": "https://你的服务器/packs/cjc-pack-1.2.0.zip",
      "pack_hash": "sha256:abc123...",
      "mc_version": "1.20.1",
      "loader": {
        "type": "fabric",
        "version": "0.15.11"
      }
    }

5. 上传 zip + server.json 到服务器
```

### 1.3 版本命名

每次发布新版本时：
```
cjc-pack-1.0.0.zip    ← 初始版本
cjc-pack-1.1.0.zip    ← 加模组/改配置
cjc-pack-1.2.0.zip    ← 更新模组
```

服务端保留所有历史版本（客户端可以做增量对比）。

## 二、更新器处理 zip 的流程

### 2.1 同步算法

```go
func syncFromZip(cfg *ServerConfig, mcDir string, versionName string) error {
    zipURL := cfg.PackURL
    zipHash := cfg.PackHash
    packDir := filepath.Join(mcDir, "versions", versionName)
    
    // ── Step 1: 下载 zip 包 ──
    zipPath := filepath.Join(packDir, ".pack.zip")
    if err := downloadFile(zipURL, zipPath, zipHash); err != nil {
        return fmt.Errorf("下载整合包失败: %w", err)
    }
    
    // ── Step 2: 读取 zip 清单 ──
    reader, _ := zip.OpenReader(zipPath)
    defer reader.Close()
    
    zipManifest := make(map[string]ZipEntry)
    for _, f := range reader.File {
        zipManifest[f.Name] = ZipEntry{
            Name: f.Name,
            Size: f.UncompressedSize64,
            CRC32: f.CRC32,
            IsDir: f.FileInfo().IsDir(),
        }
    }
    
    // ── Step 3: 对比本地已同步的文件 ──
    // 读取上一次同步的 zip hash（记录在 manifest.json 中）
    prevManifest := loadPackManifest(packDir)
    
    if prevManifest != nil && prevManifest.ZipHash == zipHash {
        // hash 没变，说明 zip 没更新
        fmt.Println("  整合包已是最新")
        return nil
    }
    
    // ── Step 4: 确定要更新的文件 ──
    // 规则：zip 里有的文件 → 覆盖
    //       zip 里没有、本地有的文件 → 不动（用户自定义的保留）
    //       但 repair 模式下 → 删除本地多余的
    
    for _, f := range reader.File {
        if f.FileInfo().IsDir() {
            continue
        }
        
        targetPath := filepath.Join(packDir, f.Name)
        
        // 检查本地文件是否与 zip 中一致
        localHash := computeFileHash(targetPath)
        zipContent := readZipEntry(f)
        zipHash := sha256Hash(zipContent)
        
        if localHash == zipHash {
            continue // 相同，跳过
        }
        
        // 写入文件
        os.MkdirAll(filepath.Dir(targetPath), 0755)
        os.WriteFile(targetPath, zipContent, 0644)
        fmt.Printf("  📄 ~ %s\n", f.Name)
    }
    
    // ── Step 5: 保存本次同步记录 ──
    savePackManifest(packDir, &PackManifest{
        ZipHash: zipHash,
        Version: cfg.Version,
        SyncedAt: time.Now(),
    })
    
    fmt.Println("  整合包同步完成")
    return nil
}
```

### 2.2 同步策略

| 模式 | zip 中有 | zip 中无 | 适用场景 |
|---|---|---|---|
| **sync（默认）** | 覆盖 | 保留本地 | 日常更新，用户加的额外模组不动 |
| **repair** | 覆盖 | 删除 | 修复模式，恢复纯净状态 |

### 2.3 zip 中的目录映射

```go
var zipDirMapping = map[string]string{
```

## 三、增量更新的实现

### 3.1 下载新 zip ≠ 全量解压

每次从服务器拉新 zip 全量解压太浪费带宽和时间。优化方案：

```go
// 方案：服务端保留旧版本的 zip，客户端通过 HTTP Range 请求只获取差异
// 但这太复杂了。实际可行的方案：
```

改为**本地仓库方案**（已经在 `本地版本仓库与增量同步.md` 中设计了）：

```go
// 增量同步流程：
// 1. 下载最新 zip → 放到 starter_repo/files/ 缓存
// 2. 解压 zip 到临时目录
// 3. 对比临时目录 vs 当前版本目录
//    → 新增的：复制
//    → 变动的：覆盖
//    → 删除的：按 sync/repair 模式处理
// 4. 删除临时目录
// 5. 记录版本快照

func syncPackIncremental(cfg *ServerConfig, versionName string, repo *LocalRepo) error {
    // 下载 zip 到缓存
    zipPath, err := repo.Cache.Download(cfg.PackURL, cfg.PackHash)
    if err != nil {
        return err
    }
    
    // 解压到临时目录
    tmpDir, _ := os.MkdirTemp("", "pack-extract")
    defer os.RemoveAll(tmpDir)
    
    extractZip(zipPath, tmpDir)
    
    // 对比
    remoteFiles := scanDir(tmpDir)
    localFiles := scanDir(versionDir(versionName))
    
    changes := diffFiles(remoteFiles, localFiles)
    
    // 按变更类型处理
    for _, c := range changes {
        switch c.Action {
        case "add":
            copyFile(c.Src, c.Dst)
        case "update":
            copyFile(c.Src, c.Dst)
        case "delete":
            if cfg.RepairMode {
                os.Remove(c.Dst)
            }
        }
    }
    
    return nil
}
```

### 3.2 带宽估算

```
典型整合包大小：30~80 MB（模组 + 配置）

首次同步：下载完整 zip，30~80 MB
后续每次版本更新：下载完整新 zip，30~80 MB
  但旧 zip 的缓存文件可以复用 → 实际差异下载量可能只有 <10 MB

一个月更新 4 次，zip 体积不在本地缓存（只保留 hash 记录）：
  无缓存：80 × 4 = 320 MB
  有缓存（files/ 保留旧模组副本）：80 + 10×3 = 110 MB
```

## 四、server.json 完整用例

```json
{
  "version": "1.2.0",
  "mc_version": "1.20.1",
  "loader": {
    "type": "fabric",
    "version": "0.15.11"
  },
  "pack": {
    "url": "https://你的服务器/packs/cjc-pack-1.2.0.zip",
    "hash": "sha256:abc123..."
  }
}
```

更新器拿到 `pack.url` 后做的事情：
```
1. 下载 zip
2. 校验 hash
3. 解压到 .minecraft/versions/{version_name}/
4. 下载 MC 本体（client.jar + assets + libraries）
5. 安装 Fabric
6. 写入 launcher_profiles.json + PCL.ini
7. 注册到 PCL2 注册表
```

## 五、和之前 PCL2 导出的关系

| 方式 | 产出物 | 优点 | 缺点 |
|---|---|---|---|
| PCL2 导出 .mrpack | `modrinth.index.json` + overrides/ | 模组只保留 URL，包体小 | 需要联网下载模组 |
| 直接 zip | `mods/`, `config/`, ... | 开袋即食，不依赖网络 | 包体大，二次分发需注意协议 |

**你们的 zip 方案更好**——因为整合包是给自家/特定用户用的，不公开发布，包体大小不是问题，关键是开箱即用、零网络依赖。
