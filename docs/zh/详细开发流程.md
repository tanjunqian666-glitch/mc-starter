## P1：MC 版本下载 + Asset 管理

**目标**：能从 Mojang API / 镜像拉取指定版本的 MC，正确下载资产文件

### 任务清单

#### 1.1 版本清单同步
- [ ] 实现 Mojang 版本清单 API 解析
- [ ] 从 mirror 自动选择数据源
- [ ] 版本列表缓存到本地（`updater/versions/version_manifest.json`）
- [ ] 按版本号匹配 `server.json` 指定的 MC 版本

```go
type VersionManifest struct {
    Latest struct {
        Release  string `json:"release"`
        Snapshot string `json:"snapshot"`
    } `json:"latest"`
    Versions []struct {
        ID          string `json:"id"`
        Type        string `json:"type"`
        URL         string `json:"url"`
        ReleaseTime string `json:"releaseTime"`
    } `json:"versions"`
}
```

#### 1.2 版本 Jar 下载
- [ ] 拉取指定版本的 `version.json`（包含下载地址 + 依赖列表）
- [ ] 下载 `client.jar` 到 `.minecraft/versions/<version>/<version>.jar`
- [ ] 下载 `client-mappings.txt`（可选的，调试用）
- [ ] 下载 `server.jar`（可选的）

#### 1.3 Asset 索引同步
- [ ] 解析 `version.json` 中的 asset 索引 URL
- [ ] 下载 assets 索引文件到 `.minecraft/assets/indexes/<version>.json`
- [ ] 解析索引：`objects` 字段列出所有需要的文件

#### 1.4 Asset 文件下载
- [ ] 下载目录：`.minecraft/assets/objects/<hash[0:2]>/<hash>`
- [ ] 只下载索引中标记为 `hash` 的文件
- [ ] 批量下载时控制并发（4 个 goroutine）
- [ ] 校验每个文件的 SHA1

#### 1.5 Libraries 下载
- [ ] 解析 `version.json` 中的 `libraries` 列表
- [ ] 下载每个 library jar 到 `.minecraft/libraries/`
- [ ] 规则：
  ```go
  // package edu.uci.arc:arcanear-api:1.3.0-preview3
  // → 路径：libraries/edu/uci/arc/arcanear-api/1.3.0-preview3/arccanear-api-1.3.0-preview3.jar
  ```
- [ ] 排除 `natives`（不需要运行时 native 库）
- [ ] 检查 `rules` 字段（OS 匹配）

**验收标准**：`./starter sync` 后 `.minecraft` 里有完整的 versions + assets + libraries，能用官方启动器直接打开

#### 1.6 日志输出
- [ ] 显示当前同步阶段（"检查版本... / 下载 asset (15/203)..."）
- [ ] 每个文件下载后显示大小和速度
- [ ] 错误汇总（下载失败文件列表）

## P3：Java 环境检测

**目标**：自动化 Java 检测，提示用户安装/升级

### 任务清单

#### 3.1 Java 路径检测
- [ ] 按优先级查找：
  1. `local.json` 中 `java_home` 指定路径
  2. 环境变量 `JAVA_HOME` → `$JAVA_HOME/bin/java`
  3. 环境变量 `PATH` → `where java` / `which java`
  4. 注册表查询（Windows）：`HKEY_LOCAL_MACHINE\SOFTWARE\JavaSoft\`
  5. 常见安装路径（自动扫描）
- [ ] 优先选择版本兼容的 Java
- [ ] 支持 JDK 和 JRE

#### 3.2 Java 版本校验
- [ ] 执行 `java -version` 并解析输出
- [ ] 提取主版本号
- [ ] 与 `server.json` 的 `java.min_version` 对比
- [ ] 检测位数（32/64 bit）

```go
type JavaInfo struct {
    Path      string
    Version   int    // 主版本号，如 21
    FullVer   string // 完整版本号，如 "21.0.3"
    Is64Bit   bool
    IsJRE     bool
}
```

#### 3.3 提示与引导
- [ ] Java 不存在 → 输出安装引导：
```
[INFO] 未找到 Java 21+
[INFO] 下载地址: https://mirrors.tuna.tsinghua.edu.cn/Adoptium/21/jre/x64/windows/
[INFO] 你也可以设置 JAVA_HOME 环境变量指向你的 Java 安装路径
```
- [ ] Java 版本过低 → 输出升级提示
- [ ] 可选：自动下载安装（`server.json` 配置 `java.auto_install: true`）

**验收标准**：`./starter check` 能正确检测并报告 Java 状态

## P5：自更新 + 多通道

**目标**：启动器自身能自动更新，支持 stable/beta/dev 通道

### 任务清单

#### 5.1 更新检查
- [ ] 启动时请求 `{server_base}/starter/version.json`
- [ ] 对比本地版本号（semver）
- [ ] 不需要更新则静默跳过

```json
{
  "version": "1.2.0",
  "channel": "stable",
  "download_url": "https://example.com/downloads/starter-1.2.0.exe",
  "hash": "sha256:...",
  "changelog": "新增 Fabric 支持，修复若干 bug"
}
```

#### 5.2 下载新版本
- [ ] 下载新 exe 到临时目录 `starter.exe.new`
- [ ] 校验 SHA256
- [ ] 在旧 exe 同目录写入 `starter.exe.old` 备份
- [ ] 替换 `starter.exe.new` → `starter.exe`
- [ ] 重启新版本（传递原有参数）

#### 5.3 回滚
- [ ] 如果新版本启动失败，自动回滚到 `starter.exe.old`
- [ ] `starter self-update --rollback` 手动回滚

#### 5.4 多通道
- [ ] `channel.json` 定义通道：
```json
{
  "channel": "stable"  // 或 "beta" / "dev"
}
```
- [ ] 不同通道的 `server.json` 地址不同
- [ ] 用户通过 `server.json` 配置通道切换
- [ ] 通道信息展示在版本命令中

**验收标准**：`./starter self-update` 能下载替换自身并重启

## 接口设计汇总

### CLI 命令参考

```
starter init [--config ./local.json]
  → 初始化配置，生成默认 local.json

starter check [--config ./local.json]
  → 检查 Java、配置文件完整性
  → 输出 JSON 格式报告

  → 如果 server.json 不存在则报错提示先配置
```

## 关键约定

1. **所有网络请求优先使用 mirror**，只有 mirror 列表最后一个为官方源
2. **不写死路径**，所有路径通过 config 获取
3. **所有下载校验 hash**，校验失败重试一次后跳过并报错
4. **同步流程可被 Ctrl+C 中断**，下次运行从断点继续
5. **日志分级**：DEBUG < INFO < WARN < ERROR，默认 INFO
6. **命令行输出不要花哨**，只输出关键信息，避免扰乱 CI/脚本调用

## 开发顺序建议

如果时间有限，推荐精简开发路径：

```
P0 (CLI+配置+下载器) → 必做，是一切的基础
  └→ P1 (版本下载+仓库) → 必做，核心功能
     └→ P2 (Fabric libraries+修复栈) → 同步 + 容错
        └→ P4 (启动器兼容) → 输出目录给 PCL2/HMCL
           └→ P3 (Java检测) → 提示用户
              └→ P5 (自更新) → 分发时再补
```

**最快能用起来的路径（约 5-6 天）**：P0 → P1 → P2.1-2.2（Fabric libraries）→ P4，到 P4 时已经能 `./starter sync` 输出完整目录给 PCL2 直接用了。

## P1.9 增量同步（已实现）

**日期**：2026-04-29 | **产出**：`internal/launcher/incr_sync.go` + `incr_sync_test.go`

### 目标
将 CacheStore（P1.8）和 LocalRepo（P1.7）整合进 sync 命令，实现增量同步。

### 做了什么
- `IncrementalSync` 组件：桥接 CacheStore + LocalRepo
- sync asset 下载：三级检查（磁盘→CacheStore→下载+缓存）
- sync library 下载：`ConsumeLibraryFiles` 分流缓存命中 vs 需下载
- sync client.jar：缓存加速
- sync 收尾：自动创建/更新 repo 快照，增量差异检测

### 测试
- 14 个新增测试，全 66 个通过
- 覆盖：缓存命中、快照创建、差异计算、库文件分流、孤缓存清理

### 踩坑记录
1. **ScanDirectory 路径不匹配**：`DiffSinceSnapshot` 中 `ScanDirectory` 生成的 path key 格式为 `sodium.jar`，但 snapshot manifest 中的 key 是 `mods/sodium.jar`。修复：手动 `filepath.Walk` + `dir + "/" + info.Name()` 前缀拼接
2. **copyFile 未导出**：`repo.go` 的 `copyFile` 小写，`main.go` 引用不了。修复：改成 `os.ReadFile + os.WriteFile`
3. **refs 未使用**：`CleanOrphaned` 收集了 `refs` 但没传进 `CleanOptions.KeepHashes`。修复：正确传递
