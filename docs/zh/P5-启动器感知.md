# P5 启动器感知设计

> 更新：2026-05-01 · 重构目标：从"启动器兼容"改为"启动器感知"

## 一、背景修正

~~P5 启动器兼容~~ → **P5 启动器感知 (Launcher Awareness)**

本设计被重写，原因：
- 官方启动器不开源，无法兼容
- HMCL 优先级低，暂不投入
- 核心目标不是"让启动器兼容"我们，而是**我们感知启动器的存在**，利用启动器的已有信息

## 二、核心目标

| # | 目标 | 说明 |
|---|------|------|
| 1 | **找到启动器** | 自动检测 PCL2 在用户电脑上的位置 |
| 2 | **拿到 .minecraft 目录** | 从启动器配置文件（PCL.ini）读取 LaunchFolders，拿到用户惯用的 .minecraft 路径 |
| 3 | **确认"这是我们的包"** | 通过 sentinel 文件（`starter_repo/repo.json`）标记已托管的目录，避免误接管无关的 MC 客户端 |
| 4 | **只在需要时触发检测** | GUI 首次向导、GUI 设置弹窗「🔍」按钮、CLI `starter pcl detect` / `starter check` / `starter run` |

## 三、接管鉴别逻辑

### 3.1 问题

用户电脑上可能有多个 .minecraft 目录，每个目录里可能有多个版本和整合包。
怎么区分哪个是"我们"已经托管过的，哪个是单纯的普通 MC 客户端？

### 3.2 方案：sentinel 文件标记

每个被 mc-starter 托管过的目录下都有一个隐藏的标记文件 `starter_repo/repo.json`。

```
.minecraft/
├── starter_repo/              ← ✅ 有 repo.json = 被我们托管过
│   ├── repo.json              ← 标记文件本体
│   ├── snapshots/             ← 版本快照
│   └── files/                 ← 文件缓存
│
└── packs/
    └── main-pack/
        ├── mods/
        ├── config/
        └── .starter-pack.json  ← 可选：包级别标记
```

### 3.3 检测流程

```
输入：扫描到的候选 .minecraft 目录路径

Step 1 — 查 sentinel
  检查 {dir}/starter_repo/repo.json 是否存在且有效
  ├─ 存在 → 这是我们的目录，已托管过 → ✅ 直接接管
  └─ 不存在 → Step 2

Step 2 — 包名匹配
  从服务端拉包列表，看包名是否匹配本地目录名
  例：服务端有 pack "main-pack"
  扫描 {dir}/packs/main-pack/ 是否存在
  ├─ 存在 → 匹配，但还没标记 → 自动创建 starter_repo → 接管
  └─ 不存在 → Step 3

Step 3 — 兜底
  {dir} 只是一个普通的 MC 客户端（有 versions/）
  → 不接管，返回空（留给首次全量同步时创建新标记）
```

### 3.5 `detectMinecraftDir` 改造

```go
func detectMinecraftDir() string {
    // Step 1: 先读 PCL 配置，拿到所有 .minecraft 目录
    dirs := getPCLMinecraftDirs()
    for _, dir := range dirs {
        if launcher.IsManaged(dir) {
            return dir  // ✅ 精准命中已托管的
        }
    }

    // Step 2: 回退到默认路径
    for _, dir := range defaultPaths {
        if launcher.IsManaged(dir) {
            return dir
        }
    }

    // Step 3: 都没标记过，扫描默认路径找 MC
    for _, dir := range defaultPaths {
        if _, err := os.Stat(filepath.Join(dir, "versions")); err == nil {
            return dir  // ⚠️ 有 MC 但没被托管 → 接入后自动创建 sentinel
        }
    }
    return ""
}
```

### 3.6 多目录场景：如何处理多个符合条件的 .minecraft

#### 场景

用户电脑上可能有多个 .minecraft 目录，每个都被 `starter_repo/` 标记为已托管：

```
C:/Users/鸽鸽/AppData/Roaming/.minecraft       ← 主服整合包 A v1.0.0
D:/Games/Minecraft/.minecraft                   ← 小游戏服整合包 B v0.8.0
E:/整合包/我的世界启动器/.minecraft              ← 之前手动装的额外服 C v0.5.0
```

也可能有些目录有 .minecraft/versions/ 但没有任何 starter_repo 标记。

也可能同一个包名出现在多个目录下（比如在两个地方装了同一个服的包）：
```
C:/Users/鸽鸽/AppData/Roaming/.minecraft/packs/main-pack/  ← v1.0.0
D:/Games/Minecraft/.minecraft/packs/main-pack/              ← v1.0.0（副本）
```

#### 设计目标

**核心原则：服务端管什么，客户端就接什么。**

- 服务端 API 返回的包列表：`packA`（primary）、`packB`、`packC`
- 客户端在每个被扫描到的 .minecraft 目录下，找 `packs/{packName}/`：
  - 找到 `packs/packA/` → 这个目录管 packA
  - 找到 `packs/packB/` → 这个目录管 packB
- **一个包可以有多个目录副本**（用户在不同的地方装了同一整合包），但**客户端对一个包只认一个目录**

#### 设计变更：MinecraftDirs 多值化

当前 `LocalConfig.MinecraftDir` 是单 string，改成多路径字典：

```go
type LocalConfig struct {
    MinecraftDirs map[string]string  `json:"minecraft_dirs,omitempty"` // key=包名, val=.minecraft路径
    // 旧字段保留兼容（读写时自动迁移）
    MinecraftDir string              `json:"minecraft_dir,omitempty"`  // 已废弃，读时写入 MinecraftDirs["_default"]
}
```

兼容策略：
- 读完 `local.json`，如果 `MinecraftDir` 有值且 `MinecraftDirs` 为空，自动迁移到 `MinecraftDirs["_default"]`
- 写入时只写 `MinecraftDirs`，不再写旧字段

#### 检测流程（多目录 + 多副本版）

```
PCL 检测 → 拿到 N 个候选 .minecraft 目录
              ↓
        对每个目录：
          read starter_repo/repo.json
              ├─ 有 repo.json → 读取 managedPackNames（snapshots 里存的包名列表）
              │                    加入 managedDirs[packName] = append(...)
              └─ 无标记但有 versions/ → 加入 rawDirs（未托管但有 MC）
              ↓
        冲突检测：同一个 packName 出现在多个目录
              ↓
        给出结果：
          ┌───────────────────────────────────────┐
          │ IsManagedDirs()                       │
          │  → []ManagedMCDir                     │
          │     [{path, packs[], latestVer}]       │
          │  → []ConflictEntry                    │
          │     [{packName, paths[], versions[]}]  │
          └───────────────────────────────────────┘
```

#### 副本冲突解决策略

当同一个包名出现在多个托管目录时：

| 策略 | 适用场景 | 做法 |
|------|----------|------|
| **取最新的** | 全自动/无交互 | 比较 `repo.json.UpdatedAt` + 包 `LocalVersion`，选最晚更新的目录 |
| **取已记录的** | 已有配置 | `localCfg.MinecraftDirs[packName]` 已存在 → 用已记录的路径 |
| **用户选** | GUI 向导/CLI | 列出所有副本让用户挑 |

优先级：**已记录 > 最新 > 用户选**

实际逻辑（`selector`）：
```
func resolveDir(packName string, candidates []ManagedMCDir) string:
  1. if localCfg.MinecraftDirs[packName] in candidates → 返回它
  2. 按 repo.json.UpdatedAt 排序，选最新的
  3. 如果平手 → 选目录名更短的（越深层越不像主目录）
```

#### 合并到 `detectMinecraftDir` 改造

```go
func detectMinecraftDir(cfg *model.LocalConfig) []ManagedMCDir {
    dirs := getCandidateDirs() // 从 PCL 配置 + 默认路径收集

    managed := make(map[string][]ManagedMCDir) // packName → 候选列表
    rawDirs := []string{}                      // 未托管目录

    for _, dir := range dirs {
        if rm, err := readRepoMeta(dir); err == nil {
            for _, packName := range rm.ManagedPacks {
                managed[packName] = append(managed[packName], ManagedMCDir{
                    Path: dir,
                    Packs: rm.ManagedPacks,
                    UpdatedAt: rm.UpdatedAt,
                })
            }
        } else {
            rawDirs = append(rawDirs, dir)
        }
    }

    return resolveAllDirs(managed, rawDirs, cfg)
}
```

### 3.7 同名包副本冲突解决策略

#### 场景 A：同包名、不同目录（已在上节覆盖）

```
C:/.../.minecraft1/packs/main-pack/  ← v1.0.0
D:/.../.minecraft2/packs/main-pack/  ← v1.0.0
```

→ 用 `resolveDir()` 已记录 > 最新 > 用户选

#### 场景 B：同目录、不同包名

**同一份东西，只是因为用户装了两遍、手动重命名、或者用中文名又装了一次，导致同一个 .minecraft/packs/ 下有多个长得像的目录：**

```
同一个 .minecraft/packs/ 下：
  main-pack/            ← 服务端定义的正式包名
  main-pack-copy/       ← 用户手动复制了一份
  主服整合包/            ← 中文名，实际是同一个东西
  extra-content/        ← 另一个服，真的不同
```

**关键问题：服务端 API 只发来 `main-pack`，但我们不知道 `main-pack-copy` 和 `主服整合包` 是不是同一个东西。**

#### 解决方案

我们**无法 100% 确定**一个包是不是副本——不能靠猜文件名。所以策略分两层：

**第一层（确定性匹配）：包名完全一致**
```
服务端包名  main-pack
        ↓ 完全匹配
本地 packs/main-pack/    ← 就是它
```

**第二层（嫌疑检测 + 提示）：扫描相似包名，但**不自动接管**

```
检测到服务端包 "main-pack"
检查本地 packs/ 目录：
  ✅ main-pack/           ← 完全匹配，接管
  ⚠️ main-pack-copy/     ← 包名相似（含相同前缀）
  ⚠️ 主服整合包/          ← 无法判断

处理：
  main-pack → 正常接管
  main-pack-copy / 主服整合包 → 列在"疑似副本"列表
    → GUI 设置页展示："注意：发现以下未托管的包目录..."
    → CLI 不处理
    → 用户如果手动删了 packs/ 下不要的目录，我们不管
```

#### 相似度判断规则

什么时候算"疑似副本"：

1. **前缀匹配**：`main-pack-copy` 包含 `main-pack` 前缀 + 额外后缀 → 疑似
2. **中英匹配**：包名 `main-pack` + 本地目录 `主服整合包` → 人眼看得出，程序判断不了 → 不处理
3. **完全一致**：`main-pack` == `main-pack` → 确定匹配

**核心原则：宁可漏过，不可误吞。**

- 包名完全匹配 → 接管
- 包名不同但内容相似 → 不接管，仅提示
- 用户想用其他目录 → 手动在设置里指定

#### 实现

```go
// FindSuspectedDuplicates 扫描本地 packs/ 下疑似服务端指定包的副本
// 返回疑似副本目录名列表（不含完全匹配的那个）
func FindSuspectedDuplicates(packsDir, serverPackName string) []string {
    entries, err := os.ReadDir(packsDir)
    if err != nil {
        return nil
    }

    var suspects []string
    prefix := strings.ToLower(serverPackName)

    for _, e := range entries {
        if !e.IsDir() {
            continue
        }
        name := e.Name()
        if name == serverPackName {
            continue // 完全匹配，不是副本
        }
        // 前缀匹配：main-pack-copy → 含 main-pack 前缀
        lowered := strings.ToLower(name)
        if strings.HasPrefix(lowered, prefix) {
            suspects = append(suspects, name)
            continue
        }
        // 不允许包名包含空格后缀？（如 "main-pack (copy)"）
        // 这个太发散，不自动判断，交给用户
    }
    return suspects
}
```

只做前缀匹配，不做中英文/语义匹配。

### 3.9 `RepoMeta` 需要加的字段

```go
// RepoMeta 仓库元信息（当前已有）
type RepoMeta struct {
    Version        int       `json:"version"`
    MCVersion      string    `json:"mc_version"`
    Snapshots      []string  `json:"snapshots"`
    LatestSnapshot string    `json:"latest_snapshot"`
    TotalCached    int64     `json:"total_cached"`
    TotalCacheSize int64     `json:"total_cache_size"`
    CreatedAt      time.Time `json:"created_at"`
    UpdatedAt      time.Time `json:"updated_at"`
}

// 新增字段
    ManagedPacks  []string  `json:"managed_packs,omitempty"` // 本目录下管理的包名列表
```

这个字段在每次更新/同步后刷新，存的是 `.minecraft/packs/` 下实际存在的包目录名。
作用是让 `IsManagedDirs()` 只需要读 `repo.json` 就能知道这个目录管了哪些包，不用再去扫 `packs/` 目录。

| 场景 | 行为 | 变化 |
|------|------|------|
| 单目录单包（最常见） | 自动检测接管，无需任何选择 | 无变化 |
| 服务器 PCL.ini 指向固定目录 | 读 PCL 配置直接拿到目录 | 更精准了 |
| 用户在多个目录装了同一整合包 | 选已记录的 / 选最新 / 让用户选 | 新增冲突提示 |
| 用户第一次用 | 首次同步后自动创建 sentinel 和 repo，后续就是"已托管" | 无变化 |

### 4.2 发现的问题

- **`detectLauncher()` 硬编码文件名**：`filepath.Join(result.PCLDir, "PCL2.exe")` → 应直接返回 `result.Path`
- **`detectMinecraftDir()` 不会读 PCL 配置**：只扫硬编码路径，没有走 `PCLDetection.ReadPCLConfig()`
- **`starter pcl` 子命令空壳**：`detect` 和 `path` 都是 `"not yet implemented"`
- **`starter check` 没调 PCL 检测**
- **`starter run` 检测了 PCL2 但不写回 `localCfg`**：每次都要重新检测
- **缺少 `IsManaged()`**：没有鉴别"这是我们的包"的入口
- **`finder.go` 和 `pcl_detect.go` 有重复**：`VersionFinder.findViaPCL` 内部又做了 `FindPCL2()` + `ReadPCLConfig()`
- **代码名写死 "PCL"**：没有泛化接口，未来加 HMCL 需重构（当前不需要，但注意约束）

### 4.3 不需要做的

- HMCL 检测（暂不投入）
- 官方启动器兼容（不开源，无法做）
- 多启动器切换 UI（裸启动/PCL2 足够）

### 测试用例

| 场景 | 期望 |
|------|------|
| 有 PCL2，PCL.ini 有 LaunchFolders | 识别 PCL2 + 读出所有 .minecraft 目录 |
| 扫描到多个 .minecraft 目录，其中部分有 `starter_repo/` 标记 | GUI 向导 MC 目录下拉框列出所有目录，[已托管]标注，默认选已记录的 |
| 同包名出现在两个托管目录下 | 下拉框候选重复，默认选已记录的/最新的，用户可切 |
| 同目录下 `main-pack` 和 `main-pack-copy` 同时存在 | `main-pack` 正常接管，`main-pack-copy` 列在"疑似副本"提示里 |
| GUI 设置勾选副版本 | 副版本目录下拉框出现，候选项同主版本 |
| GUI 设置取消副版本勾选 | 目录下拉框隐藏，但保留已选路径 |
| `starter pcl detect` 多目录 | 展示序号列表，每行标注 [已托管/候选] 及管理的包名 |
| `starter pcl set-dir main-pack 2` | 写入 `MinecraftDirs["main-pack"]` = 序号2的路径 |
| 旧配置有 `MinecraftDir` 无 `MinecraftDirs` | 自动迁移到 `MinecraftDirs["_default"]` |
| 包未指定独立目录 | 回退到 `MinecraftDirs["_default"]` |
